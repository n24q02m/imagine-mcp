## 2024-05-18 - Optimize default_provider_for lookups
**Learning:** `default_provider_for` in `src/imagine_mcp/models.py` performs an O(N log N) filtering and sorting of `MODELS` on every call. Since the parameter space (`action`, `media`, `tier`) is very small and bounded, caching the result yields a ~25x performance improvement. Standard lru_cache works perfectly for this use case.
**Action:** Use `@functools.lru_cache(maxsize=32)` to optimize functions that perform expensive queries or sorting over static lists when their parameter space is small and bounded.
## 2024-05-18 - Optimize media type detection concurrency
**Learning:** In `dispatch_understand`, media types for URLs were being detected sequentially (`media_types = [detect_media_type(u) for u in media_urls]`). Since HTTP calls via `detect_media_type` are I/O bound, this creates an O(N) network bottleneck. By wrapping `detect_media_type` with a `ThreadPoolExecutor.map`, we can fetch all media types concurrently, effectively reducing the latency to O(1). Critically, wrapping the executor map result in `list()` is required to evaluate the generator and ensure the first encountered exception is raised immediately, preserving fail-fast behavior.
**Action:** When performing independent I/O bound operations over a collection (like HTTP HEAD requests), use `ThreadPoolExecutor.map` wrapped in `list()` to evaluate the generator and preserve fail-fast error handling.
